# 要件 → GitHub Issues

ユーザーの要件を受け取り、コードベース調査・要件詳細化・タスク分解を経て GitHub Issues を一括作成する。コードは書かない。

## フェーズ概要

| フェーズ | 内容 |
|---------|------|
| Phase 0 | 要件受け取り（不足なら 1 問 1 答で引き出す） |
| Phase 1 | コードベース調査（Explore サブエージェント） |
| Phase 2 | 要件詳細化（1 問 1 答で曖昧点を解消） |
| Phase 3 | タスク分解 → ユーザー承認 |
| Phase 4 | GitHub Issue 一括作成 |

---

## Phase 0: 要件受け取り

`$ARGUMENTS` を確認する。目的・対象・完了条件が揃っていれば Phase 1 へ進む。

不足している場合は `AskUserQuestion` で 1 問だけ問う。

**重要: `AskUserQuestion` を呼ぶ前に必ず ToolSearch で `select:AskUserQuestion` をフェッチすること。**

確認すべき観点（揃い次第 Phase 1 へ）：

- **目的**: 何を達成したいか
- **対象**: 影響するページ・機能・データ
- **完了条件**: 何ができたら「終わり」か

1 ターンに 1 質問。複数の観点を 1 つの質問に詰め込まない。

---

## Phase 1: コードベース調査

`Agent` ツール（`subagent_type=Explore`）を使い、要件に関連するコードを調査する。

調査結果はメイン会話に直接持ち込まない。**要約（3〜5 行）と出力パスだけをメイン会話に返させる**（コンテキスト保護）。

### サブエージェントへの指示テンプレート

> 以下の要件に関連するコードを調査してください：**{要件概要}**
>
> 調査観点：
> - 要件に関連する既存ファイル・関数・コンポーネント
> - 類似実装・重複する機能の有無
> - 関連するデータ型・インターフェース（`#shared/types/` など）
> - 影響を受けそうな Composable・ページ
>
> 結果を `/tmp/req-to-issues-{timestamp}/grounding.md` に書き出してください。
> 書式：ファイルパスと対象箇所を引用（1 ファイルにつき 3〜5 行）。
> メイン会話には 3〜5 行の要約と `/tmp/...` パスのみ返してください。

---

## Phase 2: 要件詳細化

Phase 1 の調査結果を踏まえ、曖昧な点を `AskUserQuestion` で 1 問 1 答で確認する。

**`AskUserQuestion` を呼ぶ前に必ず ToolSearch で `select:AskUserQuestion` をフェッチすること。**

確認が必要な観点（すべて揃うまで続ける）：

- **スコープ**: 何をする / しない
- **受け入れ条件**: 完了の定義
- **前提・制約**: 既存コードの変更可否、UI の変更範囲など

確認が完了したら、以下のサマリーを出力してユーザーに確認を取る。**ユーザーの同意なしに Phase 3 へ進まない。**

```
## 要件サマリー

- **スコープ**: ...（する / しないの境界線）
- **受け入れ条件**: ...（完了の定義）
- **前提・制約**: ...
```

---

## Phase 3: タスク分解

詳細化された要件を、**1 Issue で実装できる粒度**のタスクに分解する。

### 各タスクに含める情報

- **タイトル**（72 文字以内）
- **目的**（1〜2 文）
- **対象ファイル**（既存ファイルがある場合）
- **受け入れ条件**（チェックボックス形式）
- **依存関係**（実装順序が必要な場合のみ）

### タスク一覧の提示フォーマット

```
## タスク一覧（{N} 件）

### 1. {タイトル}
**目的**: ...
**対象ファイル**: `app/...`
**受け入れ条件**:
- [ ] ...
- [ ] ...
**依存**: なし / #{n} の完了後

---
```

タスク一覧を提示した後、**「この内容で Issue を作成してよいですか？」と確認してから Phase 4 へ進む。**

---

## Phase 4: GitHub Issue 作成

ユーザー承認後、`gh api` で各 Issue を順番に作成する。`gh issue create` は使わない。

### リポジトリ情報の取得

```bash
gh repo view --json owner,name --jq '"repos/\(.owner.login)/\(.name)"'
```

### Issue 作成コマンド

```bash
gh api repos/{owner}/{repo}/issues \
  -X POST \
  -f title="{タイトル}" \
  -f body="{本文}" \
  --jq '{number, html_url}'
```

### Issue 本文テンプレート

```markdown
## 目的
{目的 1〜2 文}

## 対象ファイル
- `{ファイルパス}`

## 受け入れ条件
- [ ] {条件1}
- [ ] {条件2}

## 依存関係
なし / #{n} の完了後
```

### 完了後の出力

全 Issue 作成後、番号と URL の一覧を出力する：

```
## 作成した Issues

| # | タイトル | URL |
|---|--------|-----|
| #{n} | {タイトル} | {URL} |
```

---

## 制約

- Issue 作成前に必ずユーザー確認を取ること
- コードは書かない（調査・整理・分解のみ）
- 1 ターンに 1 質問まで（AskUserQuestion を使い、複数質問を詰め込まない）
- 重い調査はサブエージェントへ委譲してメイン会話のコンテキストを守る
- AskUserQuestion を呼ぶ前に必ず `ToolSearch select:AskUserQuestion` でスキーマをフェッチする
